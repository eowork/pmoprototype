RISK-058: Rank Irrelevance in CRUD
───────────────────────────────────
Threat:     Senior Staff (rank 50) and Junior Staff (rank 70) have identical
            CRUD permissions, making rank appear decorative.
Resolution: GOVERNANCE DECISION — Rank is approval-authority only, not CRUD-tier.
            This is documented as BY DESIGN. CRUD is role-based (SuperAdmin,
            Admin, Staff, Viewer). Rank influences approval chains and user
            hierarchy management only.
Status:     ✅ RESOLVED (Phase P)

RISK-059: No Centralized Backend Resolver
─────────────────────────────────────────
Threat:     Permission logic scattered across 3 service files (construction,
            repairs, university-operations). Each had duplicate isAdmin(),
            isAdminFromDatabase(), and inline rank comparison code.
Resolution: Created PermissionResolverService in common/services with:
            - isAdmin(user) — JWT role check
            - isAdminFromDatabase(userId) — Secure DB validation
            - canApproveByRank(approverId, submitterId, isSuperAdmin) — Consolidated
              rank check with optimized single DB query
            - canModifyUserByRank(actorId, targetId, isSuperAdmin) — User hierarchy
Status:     ✅ RESOLVED (Phase Q)

RISK-062: REFERENCE_DATA_MODULES Drift
──────────────────────────────────────
Threat:     Constant appeared redundant after ADMIN_ONLY_MODULES was updated.
            Purpose was unclear.
Resolution: Updated comment to clarify dual purpose:
            - ADMIN_ONLY_MODULES: Controls sidebar visibility
            - REFERENCE_DATA_MODULES: Controls form-level access (Staff can read
              contractor/funding source data for dropdowns)
Status:     ✅ RESOLVED (Phase R)

NEW RISK: Self-Approval UI Inconsistency
────────────────────────────────────────
Threat:     UI showed Approve button for own submissions. User clicks, backend
            returns 403 "Cannot approve your own submission." Confusing UX.
Resolution: Updated canApproveItem() in all 3 index pages to hide Approve button
            when item.approvalMetadata.submittedBy === currentUserId.
            SuperAdmin exempted from this check.
Status:     ✅ RESOLVED (Phase R)

RISK-060: Staff Visibility Gap (Team DRAFT)
───────────────────────────────────────────
Threat:     Staff cannot see team members' DRAFT records — only own records via
            My Drafts page. No team-level workflow visibility.
Mitigation: Out of scope for current implementation. Requires scoping decision:
            Is team-level visibility a requirement?
Status:     DEFERRED (D7)

RISK-061: JWT Missing Permission Context
────────────────────────────────────────
Threat:     rank_level and module_assignments not in JWT. Backend must re-query
            DB at every approval (two extra queries per publish).
Mitigation: PermissionResolverService.canApproveByRank() optimized to single
            query. Full JWT extension deferred as low priority.
Status:     DEFERRED (D2) — Partially mitigated

RISK-063: Permission Override Schema Limitation
───────────────────────────────────────────────
Threat:     Cannot grant action-level overrides (e.g., "Staff X can delete in
            COI"). Schema only supports binary module access (can_access: boolean).
Mitigation: Role-based CRUD sufficient per governance decision (Phase P).
            Schema migration deferred.
Status:     DEFERRED (D1)
